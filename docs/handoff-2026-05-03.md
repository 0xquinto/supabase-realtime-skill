## Methodology notes (carry across sessions)

- **Adversarial pass discipline (NEW)** — every recon/ADR ships with an explicit "is this *direction* wrong?" pass. The v1.0.0 recon got it mid-PR after pushback; future recons get it pre-PR per the saved memory rule. First risk named in any recon should be direction-failure, not execution-failure.
- **`gh api` verification on upstream PRs** — non-negotiable. Three sessions, three times the discipline mattered. The PR-titles-as-evidence shortcut is consistently wrong.
- **FAIL→fix→PASS** is for substrate-correctness ADRs. Verification-coverage ADRs (ADR-0015 shape) don't fit the pattern — say so explicitly when filing one. v1.0.0's smoke extensions are also verification-coverage shape (NOT substrate-correctness — the substrate is correct; coverage is what's expanding).
- **Branch prefixes:** `recon/` for pre-ADR research; `feat/` for ADR implementation that includes code; `fix/` for substrate-correctness fixes; `docs/` for pure-doc PRs (currency sweeps, README work). Don't put ADR numbers in branch names.
- **Recon → ADR is one-PR-per-step.** Recon merges first; ADR implementation branches off main from the merged recon. This session shipped the recon (#15); the next session does the ADR-0016 implementation.
