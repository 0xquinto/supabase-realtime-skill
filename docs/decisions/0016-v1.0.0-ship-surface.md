# ADR-0016: v1.0.0 ship surface — watch_table + subscribe_to_channel E2E smokes + MCP SDK floor

**Status:** Accepted (2026-05-03). Promoted after the smoke surface returned green on the deployed Edge Function: `watch_table` E2E `wall=5442ms delivered_n=1` (45% headroom under 12s budget; n=1 means the +5s INSERT landed first, consistent with the spike's cold-start handshake distribution), `subscribe_to_channel` E2E `wall=3110ms` (broadcast fan-out via separate sender client confirmed). All 6 `tests/smoke/edge-deploy.smoke.test.ts` blocks pass; SDK floor regression gate live in fast tests (50/50 pass).

**Date:** 2026-05-02

**Supersedes:** none. **Composes on top of:** ADR-0011 (Postgres-Changes RLS substrate), ADR-0013 (Broadcast Authorization substrate), ADR-0014 (npm `0.2.0` worked-example ship), ADR-0015 (Edge tool-routing verification + source-tree alignment).

**Related recon:** [`docs/recon/2026-05-02-v1.0.0-ship-surface-recon.md`](../recon/2026-05-02-v1.0.0-ship-surface-recon.md). The recon's six explicit decisions are committed on below; the load-bearing risk-1 mitigation (warm-up distribution spike before sized smoke) is captured in [`docs/spike-findings.md`](../spike-findings.md) § T7-Edge with two non-trivial sub-findings that re-shape the v1.0 ship.

## Context

The recon called this the "v1.0.0 ship surface": closing the four sub-items ADR-0015 deferred under § "Out of scope":
1. `watch_table` E2E smoke against the deployed function (the only tool whose substrate is the websocket leg)
2. `subscribe_to_channel` E2E smoke against a real broadcast sender (proves fan-out, not just wire-success)
3. MCP SDK pin sanity check (1.25.3 → 1.27.1+)
4. Re-verifying ADR-0011/0013 substrate correctness on the deployed function (deferred again — npm-package smoke covers it; Edge re-verification is v2 hardening)

Plus orthogonal: bumping the manifest to `n=300` (ADR-0007 design pre-staged; corpus expansion the heavy lift). Per recon Decision 4, the manifest ships as a **separate PR** — this ADR covers the smoke surface + SDK floor; the manifest is ADR-0017 (or an ADR-0007 amendment) when the n=300 corpus lands. v1.0.0 = both shipped within the same calendar week, otherwise the smoke PR is `0.3.0` and v1.0.0 stays unclaimed.

The recon's adversarial pass (§ Risk #1) flagged the load-bearing variance question: "the actual ship-blocker isn't 'does it work', it's 'does it work *reliably*'." Mitigation: a 20-trial warm-up distribution spike on Edge isolates before any smoke ships. That spike landed in [`eval/spike-edge-warmup.ts`](../../eval/spike-edge-warmup.ts) and surfaced two substrate findings that re-shape the smoke design.

## Decisions

### 1. Adopt v1.0.0 ship event = bundled-evidence completeness

**Reject** the alternatives the recon evaluated:
- (b) freeze-the-API-and-write-a-contract-doc — no demand signal; ship-as-evidence is the artifact's pattern.
- (c) upstream `supabase/agent-skills` submission as the v1.0 event — couples to external review timeline.

v1.0 ships when both this PR (smoke surface + SDK floor) and the manifest n=300 PR have landed. Same calendar week or v1.0 stays unclaimed; smoke PR ships standalone as `0.3.0` if manifest blocks. **No tagging until both are merged.**

### 2. Add `watch_table` E2E smoke against the deployed Edge Function

**Extended `tests/smoke/edge-deploy.smoke.test.ts`** with a fifth `it()` block:

- Creates a fresh temp table on the host project (gated on new `EVAL_HOST_DB_URL` env var; skips cleanly if absent — ADR-0015's smokes don't need DB)
- Applies the GRANT + RLS chain (root-cause-pinned in spike sub-finding 2 — see § "Empirical refinement"): `enable row level security` + `create policy ... using (true)` + `grant select on ... to anon, authenticated, service_role`
- Dispatches JSON-RPC `tools/call watch_table` against the deployed function URL with `timeout_ms: 12_000` (sized from spike p99 5322ms × 2.25 headroom; well within 30s smoke ceiling)
- Fires multi-INSERT schedule (+100ms / +5s / +10s) — single-shot +100ms INSERTs land pre-SUBSCRIBED and get dropped (spike sub-finding 1)
- Asserts `events.length === 1` + `closed_reason: max_events` + populated `event.new.n` content (regression check — without the GRANT+RLS chain, count would pass while row data is 401-stripped)

PASS receipt: 5/5 smoke pass against live deploy in 10.8s total runtime; watch_table case wall 638ms (warm path) on first run, 5252ms (post-warm-up) on second.

### 3. Add `subscribe_to_channel` E2E smoke with an external broadcast sender

**Extended `tests/smoke/edge-deploy.smoke.test.ts`** with a sixth `it()` block:

- External raw `createClient` sender on a public channel (separate from the function's per-request listener — CLAUDE.md "single-client channel dedup" foot-gun)
- Dispatches JSON-RPC `tools/call subscribe_to_channel` against the deployed function URL with `timeout_ms: 10_000, max_events: 1`
- Fires multi-broadcast schedule (+500ms / +3s / +6s) — single-shot +500ms broadcasts race the function's per-request subscribe handshake (~700ms-5s per spike); first one to land post-SUBSCRIBED wins
- Asserts `broadcasts.length === 1` + `closed_reason: max_events` + `payload.from === "external-sender"`

PASS receipt: 6/6 smoke pass against live deploy in 19.0s total runtime; subscribe_to_channel case wall 3084ms (the +3s broadcast landed first).

### 4. Floor the MCP SDK pin at `^1.27.1`

**Bumped `package.json`** from `"@modelcontextprotocol/sdk": "^1.0.0"` → `"^1.27.1"`. **Bumped `supabase/functions/mcp/deno.json`** from exact `npm:@modelcontextprotocol/sdk@1.29.0/...` to `npm:@modelcontextprotocol/sdk@^1.27.1/...`. Rationale per recon Decision 5: captures the `WebStandardStreamableHTTPServerTransport.onerror` fix from [PR #1580](https://github.com/modelcontextprotocol/typescript-sdk/pull/1580) (merged 2026-02-24, verified via `gh api`) without locking out additive minors.

`bun.lock` regen + `deno cache --reload` resolves to current 1.29.0 (within range; latest stable). No behavior change in the currently-deployed bundle — this widens the contract declaration; on next major bump the deployed version still resolves cleanly.

Mechanical change. No new tests; existing 49 fast tests + 6 smokes still pass.

### 5. Defer manifest v2.0.0 to a separate PR (ADR-0017 / ADR-0007 amendment)

**Out of scope** for ADR-0016. Per recon Decision 4: the smoke surface is mechanical (re-uses existing helpers); the manifest is a corpus-synthesis lift (~$30-60 + 2-3 ci-full runs at $6-9). Ship sequencing: smoke first as `0.3.0`-candidate; manifest second; v1.0.0 tag only if both land in the same calendar week.

Per recon Decision 6. "Completeness" implies "we believe everything is covered" — the artifact will never be complete in that sense (Presence stays deferred per ADR-0004; manifest n=1,000 stays a v3 question; multi-tenant fan-out re-verification on Edge stays v2 hardening). ## Empirical refinement (substrate gap surfaced by spike)

The recon's adversarial pass anticipated the *variance* risk; the spike surfaced two distinct, sharper substrate gaps the smoke must work around. Both were non-obvious without the spike.

### 1. Single-INSERT@+100ms is unsafe on Edge

First-pass spike used a single INSERT at +100ms post-`tools/call` dispatch. **17/20 trials timed out at 30s** (with both anon JWT and service_role — auth was not the variable). The per-request Edge subscribe handshake exceeds 100ms in the cold-isolate path, so the row commits pre-SUBSCRIBED and Postgres-Changes drops it. The function holds the subscription open until `timeout_ms`, then returns `closed_reason: timeout` with 0 events.

Mitigation: multi-INSERT schedule (+100ms / +5s / +10s) — at least one INSERT is guaranteed to land post-SUBSCRIBED for typical Edge cold-start budgets. v1.0.0's watch_table smoke uses this pattern (mirrors `tests/smoke/watch-table.smoke.test.ts:87-89`); single-shot designs would flake in consumer hands.

### 2. Realtime row payload requires GRANT + RLS chain (root cause pinned)

After fixing #1, every delivered event carried `new: {}` + `errors: ["Error 401: Unauthorized"]`. Diagnostic probe ([`eval/probe-edge-payload.ts`](../../eval/probe-edge-payload.ts)) decomposed across 7 variants:

| Variant | Auth | Table setup | Result |
|---|---|---|---|
| C | service_role (legacy JWT) | bare `CREATE TABLE` | events=1, `new: {}`, `errors: 401` |
| D | service_role (legacy JWT) | `+ GRANT SELECT` | events=1, **populated `new`**, `errors: null` |
| E | service_role (legacy JWT) | `+ GRANT + RLS policy using(true)` | events=1, **populated `new`** |
| F | anon (legacy JWT) | `+ GRANT SELECT` (no RLS) | **events=0** |
| G | anon (legacy JWT) | `+ GRANT + RLS policy using(true)` | events=1, **populated `new`** |
| (asym) | new `sb_secret_*` | bare | websocket protocol error 1002 |

Realtime broker authorizes the row payload separately from PostgREST. Without `GRANT SELECT` to the agent's role on the source table, the broker delivers the event metadata but replaces the row with `errors: ["Error 401: Unauthorized"]`. For service_role, GRANT alone is sufficient (RLS-bypass). For anon (the consumer contract), GRANT alone delivers zero events — RLS must be enabled with at least one permissive `select` policy for events to reach the channel at all.

This wasn't caught earlier because:
- Branches via `withBranch` get auto-grants on table creation (Supabase branch provisioning hook)
- `support_tickets` (used in ADR-0014's worked example) was created via migration which auto-grants
- `tests/smoke/watch-table.smoke.test.ts` doesn't inspect `new` content — silently passed
- Direct `sql.unsafe(create table ...)` against the host project (the spike's shape) doesn't auto-grant

Mitigation in v1.0.0:
- Spike updated to apply the chain after CREATE TABLE; re-run shows `new` populated, p99 5322ms unchanged within margin, recommendation 12s wall budget unchanged
- Smoke applies the same chain on its temp table — `events.length === 1` regression-checked alongside `event.new` content (catches future chain breaks rather than silently passing on count)
- Substrate concern surfaced in spike-findings as "consumer-facing implication": consumers driving `watch_table` from agents may not realize a table they "own" needs explicit GRANT + RLS-with-policy to receive postgres-changes row data via Realtime under anon JWT. Worth a future `references/` page revision; not blocking v1.0 (the contract is documented in spike-findings).

This is **NOT a substrate-correctness bug** — it's the standard Supabase contract. The skill's primitive doesn't enforce it; consumers configure their tables. The spike + probe just documented the contract empirically for the deployed-Edge code path.

## Falsifiable predicted effect

> **An external consumer of `supabase-realtime-skill@^0.2.0` who runs `supabase functions deploy --no-verify-jwt mcp` against a Pro project, then exercises JSON-RPC `tools/call` for `watch_table` (against a freshly-published table with the GRANT + RLS chain) and `subscribe_to_channel` (against a public channel with an external broadcast sender) on the deployed URL, observes:**
> - **`watch_table` delivers an INSERT event with populated `new` content within a 12s wall budget**
> - **`subscribe_to_channel` receives a fan-out broadcast within a 10s wall budget**
> - **The `^1.27.1` SDK floor resolves to 1.29.0 (or newer) without breaking the build**
> - **All 6 it() blocks in `tests/smoke/edge-deploy.smoke.test.ts` PASS in <25s wallclock**

Properties:
- **Smoke-test-shaped.** Same shape as ADR-0015 evidence; receipt is `tests/smoke/edge-deploy.smoke.test.ts` 6/6 PASS.
- **Not substrate-correctness.** No broken behavior to fix — the smokes verify the surface holds end-to-end. Closes the deferred-from-ADR-0015 verification gaps.
- **Falsifiable in five named directions:**
  - If a future deno.json regression drops MCP SDK below 1.27.1, transport-layer errors get silently swallowed (the `onerror` fix is the load-bearing reason for the floor; not directly tested by the smoke since we don't deliberately fault the transport).
  - If the GRANT + RLS chain regresses (e.g., chain steps re-ordered, policy syntax broken), the watch_table smoke fails on `event.new` populated assertion.
  - If the function's per-request subscribe handshake exceeds 6s on the host project (cold-isolate worse than spike measured), the subscribe_to_channel smoke times out (wall budget too tight).
  - If `tools/call subscribe_to_channel` returns 0 broadcasts despite the external sender firing, fan-out is broken at the broker (or RLS on `realtime.messages` denies receiver — public channel should be open).
  - If the spike's p99 wall budget recommendation (12s) underestimates real-consumer cold-start variance, the smoke flakes in CI but not locally; v2 hardening adds retry-envelope or longer budget.
- **Cost:** no metered cost — the host Pro project covers compute; the smokes add ~5 min of host-project DDL/INSERT time per `bun run test:smoke` run.

## Risks / Open questions

1. **Smoke wall budget headroom.** 12s for watch_table is 2.25× p99; 10s for subscribe_to_channel is similar headroom. If the host project's Edge isolate cold-starts more than measured (the spike serialized warm), CI runs may flake. Mitigation: smoke is gated on `EVAL_HOST_DB_URL` (only runs when operator invokes manually); ci-fast doesn't run online smokes.

2. **Substrate gap is consumer-facing.** The GRANT + RLS chain is the standard Supabase contract, but consumers driving `watch_table` from agents against tables they "own" may not realize the chain is required for the agent's anon JWT to receive populated row data. Not a bug in the skill; documentation gap. Worth a future `references/` page revision; not blocking v1.0.

3. **Asymmetric-key websocket failure stays open.** The new `sb_secret_*` keys (post-2025 migration on this host project) cause websocket protocol error 1002 on the Realtime handshake. Not a regression in this skill; substrate-level. Documented in spike-findings § T7-Edge sub-finding 2 table; future ADR or substrate fix from Supabase.

4. **The smoke uses `EVAL_HOST_DB_URL` for direct DDL.** That's a real DB connection from the smoke runner — operators who want to gate online smokes via CI would need to register that secret. ADR-0015's pattern (CI gates on `secrets.X` via step output) applies; not implementing CI gate in this ADR.

5. **Source-tree-vs-deployed alignment unchanged.** The deno.json change (SDK floor) doesn't require redeploy — current bundle resolves to same 1.29.0. Recommended on next major bump per ADR-0015 § Risks #4.

6. **CLAUDE.md currency drift will accumulate.** Once v1.0.0 ships, more lines will drift (versions, ADR counts, status block). The CLAUDE.md sweep-PR pattern from `b5078a0` + `043cc75` needs to stay habit. Already overdue per the prior-handoff follow-up #3.
