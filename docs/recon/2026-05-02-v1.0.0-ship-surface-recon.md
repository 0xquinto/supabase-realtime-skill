# Recon: v1.0.0 ship surface — what closes the substrate-completeness gap (2026-05-02)

Pre-ADR recon for the v1.0.0 ship event. Filed on branch `recon/v1.0.0-ship-surface`. Mirrors the shape of [`2026-05-02-edge-function-tool-routing-recon.md`](2026-05-02-edge-function-tool-routing-recon.md) (decisions-and-receipts) and [`2026-05-02-worked-example-ship-recon.md`](2026-05-02-worked-example-ship-recon.md) (delta-against-prior-state). The headline question is direction, not features.

> **What's the v1.0.0 ship event, and what evidence do we owe before we can tag it?**

The roadmap conversation surfaced three candidate framings. Restating tightly so this recon's job is clear:
- **(a)** Bundle the deferred-from-ADR-0015 smoke gaps + manifest n=300 → tag v1.0.0 as evidence-shaped completeness.
- **(b)** Freeze the surface API, write a v1.0 contract doc, then tag.
- **(c)** Submit to `supabase/agent-skills` upstream (T31) and let that submission *be* the v1.0 event.

Recon's tilt before evidence: **(a)**, because every prior ship was evidence-shaped (FAIL→PASS receipts, manifest gates, smoke 4/4 PASS). (b) writes a check the ecosystem isn't asking us to cash. (c) couples our v1.0 to someone else's review timeline. This recon's job is to validate (a) with concrete sub-decisions, or surface a reason to redirect.

## Why this recon, why now

ADR-0015 closed the tools/list + tools/call coverage gap on Edge but explicitly deferred three sub-items to the "v1.0.0 staging surface" (CLAUDE.md status block § 4 after the ADR-0015 currency sweep, plus ADR-0015 § "Out of scope"):

1. **`watch_table` E2E smoke** — exercises Postgres-Changes via Realtime websocket against the deployed Edge Function. Adds the warm-up window (~5s) to wall time. Highest-rigor of the three because it's the only tool whose substrate is the websocket leg.
2. **`subscribe_to_channel` E2E smoke against a real broadcast sender** — proves fan-out, not just wire-success. Pairs naturally with the `broadcast_to_channel` probe ADR-0015 shipped.
3. **MCP SDK pin sanity** — Supabase blessed pattern uses 1.25.3; we're on 1.29.0. ADR-0015 § "Out of scope" said "worth a sanity check, not load-bearing." External research (below) closes this question without a separate ADR.

Plus one orthogonal v1.0.0 candidate: **manifest n=300** — already designed in ADR-0007 (Status: Proposed; locked design; ships when the n=300 fixture corpus lands). At n=100, the `missed_events_ci_high_max` and `spurious_trigger_ci_high_max` cells are mechanically unreachable (Wilson upper at 0/100 ≈ 0.0370 > our 0.01/0.03 thresholds). v1.0.0 either ships n=300 or stays honest about the unreachable-by-design cells in the README.

Plus one orthogonal-and-already-flagged: **CLAUDE.md currency sweep** — user reverted the bundled edits in the ADR-0015 PR; goes on its own follow-up branch when comfortable. Out of scope for this recon.

## Internal recon

### `tests/smoke/watch-table.smoke.test.ts` — the canonical multi-insert pattern

Lines 73-89: warm-up workaround fires three INSERTs at +100ms, +5s, +10s after subscription so at least one lands post-warm-up. Wall time budget 30s (`boundedWatch` `timeout_ms: 30_000`). The smoke against a real branch (not Edge), but the pattern transposes cleanly: replace the direct `boundedWatch` call with a JSON-RPC `tools/call` to `watch_table` against the deployed function URL, with the smoke driving the INSERTs from a separate `postgres()` connection.

The branch lifecycle is the wall-time hog (~120-180s for `withBranch` to provision). The actual Realtime work is ~10-30s. For Edge smoke, we re-use the host project's `support_tickets` table (or apply a temp publication scoped to a dedicated table for the smoke) — no `withBranch` needed.

### `eval/spike-latency.ts` — the long-lived adapter pattern

Lines 1-30 document the T7 finding: ONE long-lived subscription, one warm-up INSERT, then n=20 INSERTs against the same subscription. Each trial measures `performance.now()` from INSERT commit to event receive.

The spike-latency pattern is the n=20 internal benchmark; the smoke pattern (multi-insert schedule, single trial) is the regression gate. v1.0.0 doesn't add to the spike-latency methodology — it adds the smoke that exercises the same primitive over the wire.

### Edge Function entry already supports `watch_table`

`supabase/functions/mcp/index.ts` builds the `Server` per-request with the auto-injected `SUPABASE_URL` + `SUPABASE_ANON_KEY` + `SUPABASE_DB_URL` + forwarded `Authorization`. The `watch_table` case in `src/server/server.ts:192-201` wires `makeSupabaseAdapter` with the same authToken propagation pattern `broadcast_to_channel` uses. **No source change needed for the smoke** — the function already routes `watch_table` correctly.

The single open question is whether the Edge isolate's 150s wall-time cap composes cleanly with `boundedWatch`'s `timeout_ms` cap of 120s. ADR-0015 already established the 30s margin (150 - 120 = 30) is sufficient for setup + teardown. For smoke purposes we cap at 30-45s to leave headroom for warm-up.

### `subscribe_to_channel` smoke shape — needs a separate sender

The current `broadcast_to_channel` smoke in `tests/smoke/edge-deploy.smoke.test.ts` (post ADR-0015) sends to `edge-smoke-${Date.now()}` and asserts `{success: true}` from the wire — no listener. To prove fan-out, the smoke needs:
- A separate `createClient` listener (CLAUDE.md "single-client channel dedup" gotcha: must be a separate client from the sender)
- Subscribe to the channel via the npm-package adapter, wait for SUBSCRIBED
- Issue the `tools/call broadcast_to_channel` against the deployed function
- Assert the listener receives the event within a wall-time window

This is more wiring than the existing broadcast probe but matches the smoke style for the multi-tenant tests (`tests/smoke/multi-tenant-rls.smoke.test.ts` § "broadcast end-to-end").

### Manifest n=300 — ADR-0007 already locks the design

ADR-0007 (Status: Proposed) pre-stages `manifest.json` v2.0.0 with:
- `n=300` for ci-full (replaces n=100)
- Tightened CI gates: `missed_events_ci_high_max`, `spurious_trigger_ci_high_max` thresholds achievable at n=300
- ADR-0010 amendment adds `forward_correctness_rate_min` + `..._ci_low_min` cells

What's *not* in ADR-0007 yet: the n=300 fixture corpus itself. ADR-0001 estimated $6-9 per ci-full run for n=300; corpus synthesis (the 200 net-new fixtures past the existing n=100) is a separate cost (ADR-0005's data-quality audit playbook applies — hand-curated seeds, then LLM-augmented at controlled ratios, then spot-checked).

The v1.0.0 question for the manifest: do we ship n=300 with the new corpus *now* (heavy lift, ~$30-60 corpus synthesis + 2-3 ci-full runs to validate at $6-9 each), or stay at n=100 with explicit "the two CI cells are mechanically unreachable until the corpus expansion" framing in the README?

## External research findings

Two parallel Exa probes covered the smoke-test substrate angle and the eval-corpus / SDK-pin angle. Findings synthesized below; primary sources cited inline.

### 1. Postgres-Changes E2E smoke against deployed Edge Functions is *unmapped territory*

**Headline:** No public evidence anyone runs `postgres_changes` smoke against a *deployed* Edge Function. Supabase's own integration tests (e.g., [supabase/realtime commit `02165bc`](https://github.com/supabase/realtime/commit/02165bcf869137f1bd704fc9ffebb7f0a5eea1b8), 2025-11-25) run a Deno test against the realtime container directly — function-as-realtime-client is undocumented. The [Supabase byo-mcp guide](https://supabase.com/docs/guides/functions/websockets) only covers function-as-websocket-server.

**Implication:** v1.0.0's `watch_table` Edge smoke is genuinely novel work — that's a *feature* of the artifact (worked-example-on-blessed-pattern is what the audience is looking for), not a research gap. The recon's confidence in the `watch_table` smoke shape rests on internal precedent (`tests/smoke/watch-table.smoke.test.ts`) plus Supabase's own Deno test patterns.

### 2. The warm-up bug is still live; `system`-event workaround exists but doesn't fit the MCP boundary

**Headline:** [supabase-js#2029](https://github.com/supabase/supabase-js/pull/2029) — candidate fix for the premature-`SUBSCRIBED` bug — is **OPEN, not merged** as of 2026-05-02 (verified via `gh api` per saved memory rule). The bug is unfixed at upstream HEAD. The canonical workaround documented in [supabase/realtime#282](https://github.com/supabase/realtime/issues/282) is:

```ts
channel.on('system', {}, payload => {
  if (payload.extension === 'postgres_changes' && payload.status === 'ok') {
    // NOW it's actually ready
  }
});
```

with a known foot-gun: this `system` message fires twice on initial subscription — the first must be skipped.

**Implication:** the `system`-event approach is real but **doesn't fit our MCP boundary**. The bounded primitive abstracts the channel away from the agent; exposing a `system`-ready signal to agents is a v2 ADR conversation, not a v1.0 smoke design. Stick with the multi-insert schedule pattern — same shape as `tests/smoke/watch-table.smoke.test.ts`. The pattern is robust and doesn't depend on upstream PR #2029 landing.

### 3. Supabase's own Deno integration test pattern uses busy-wait polling + `withDeadline(10000)`

**Headline:** Supabase's [`supabase/realtime` commit `02165bc`](https://github.com/supabase/realtime/commit/02165bcf869137f1bd704fc9ffebb7f0a5eea1b8) (edgurgel, 2025-11-25) is the cleanest reference for `postgres_changes` testing:

```ts
while (channelV1.state !== "joined" || channelV2.state !== "joined") {
  await sleep(0.2);
}
// then INSERT, then:
while (resultV1 == null || resultV2 == null) {
  await sleep(0.2);
}
```

with `withDeadline(..., 10000)` for a 10s wall-time cap, `sanitizeOps: false, sanitizeResources: false` to silence Deno's leak detector for websockets, and teardown `await channel.unsubscribe(); await stopClient(client);`.

**Implication:** Supabase's own test caps at 10s wall — our existing `tests/smoke/watch-table.smoke.test.ts` caps at 30s (multi-insert schedule fires at +100ms / +5s / +10s). For the Edge smoke, **30s is the right budget** — it gives 5x headroom over the warm-up + steady-state (~6-7s) without bumping the Edge isolate's 150s ceiling. The 120s `timeout_ms` cap that production agents use is for reliability, not smoke time-bounds.

### 4. MCP SDK 1.25.3 → 1.29.0 — no breaking changes; floor at `^1.27.1` for `onerror` fix

**Headline:** SDK adopted SemVer 2.0.0 with explicit [`VERSIONING.md`](https://github.com/modelcontextprotocol/typescript-sdk) policy in 1.27.1; minor bumps are additive. The 1.27.1 → 1.29.0 diff for `webStandardStreamableHttp.ts` is +23/-1 lines, all additive. 1.29.0 changelog is conservative: extensions field, `ResourceSchema.size`, stdio Windows fix, OAuth loopback port relaxation, TTL null-rejection — zero transport-layer behavioral changes.

**One real Deno-relevant fix at 1.27.1** ([PR #1580](https://github.com/modelcontextprotocol/typescript-sdk/pull/1580), **MERGED 2026-02-24** verified via `gh api`): `WebStandardStreamableHTTPServerTransport` was silently swallowing transport errors before `onerror` callback was wired. If we were on Supabase's documented 1.25.3, we'd be missing this fix.

**Implication:** the v1.0.0 move on the SDK pin is a tight floor (`^1.27.1`) without an artificial ceiling — captures the `onerror` fix that materially improves error visibility on Edge, doesn't lock us out of additive minors. Document the floor rationale in the v1.0.0 ADR; don't pin to a specific minor. **Pin sanity is mostly theater for 1.29 specifically.**

### 5. Manifest n=300 has principled defense — `arxiv:2503.01747` is the surgical cite

**Headline:** Three convergent sources support n=300 as the production-credible floor for binary correctness with Wilson CIs:
- [arxiv:2503.01747](https://arxiv.org/pdf/2503.01747) — "Wilson and Bayesian intervals achieve correct coverage at N=300 where CLT-based intervals fail at N=100 — empirical coverage 92.5% vs nominal 95%". The cleanest paper-cite for "n=100 is undercovered, n=300 is the inflection."
- [Cochran's formula](http://swept.ai/scoring-methodology) — n=385 is the textbook number for ±5% margin at 95% confidence with worst-case p=0.5. Swept AI uses this in their public scoring methodology.
- [Anthropic's eval-stats paper](https://arxiv.org/html/2411.00640v1) — recommends n≥1,000 for *paired model comparisons* at α=0.05, power=0.80, single-digit MDE. Worked example reaches n=198 with paired+clustered variance reduction.

**Implication:** n=300 is justifiable and the cheapest legible upgrade. Honest framing: "n=300 closes the CLT-coverage gap (arxiv:2503.01747); n=385 closes the ±5% margin gap (Cochran); n=1,000 enables paired model comparison (Anthropic)." For v1.0.0, n=300 is the right tier — it's evidence-shaped and citable. n=385 is "more correct" but the marginal $1-2 / run cost doesn't move the credibility needle vs n=300, and the corpus synthesis cost scales linearly. **Recommend n=300, lead with arxiv:2503.01747 as the citation.**

## Six explicit decisions ADR-0016 (or whatever) has to make

In rough order of effect:

1. **Adopt the v1.0.0 ship event = (a) bundled-evidence completeness.** Reject (b) freeze-and-write-contract (no demand signal); reject (c) upstream-submission-as-event (couples to external review timeline). v1.0.0 ships when the receipts below land. The recon's tilt holds.

2. **Add `watch_table` E2E smoke against the deployed Edge Function.** Extend `tests/smoke/edge-deploy.smoke.test.ts` with a fifth `it()` block: create a temp publication-membered table, fire the `tools/call watch_table` against the deployed function URL with `timeout_ms: 30_000` + `max_events: 1`, drive INSERTs at +100ms / +5s / +10s from a separate `postgres()` connection (mirror of `watch-table.smoke.test.ts:87-89`), assert one event arrives within wall budget. Cleanup: `drop table` + `alter publication ... drop table`. Genuinely novel work — no public prior art.

3. **Add `subscribe_to_channel` E2E smoke with a separate broadcast sender.** Extend `edge-deploy.smoke.test.ts` with a sixth `it()` block: create a separate `createClient` listener (per the CLAUDE.md "single-client channel dedup" foot-gun), subscribe to `edge-smoke-fanout-${Date.now()}`, issue `tools/call broadcast_to_channel` from the agent-side, assert the listener receives within a wall budget. Proves fan-out, not just wire-success. Pairs with the `broadcast_to_channel` probe ADR-0015 shipped.

4. **Bump the manifest to v2.0.0 with n=300.** Promote ADR-0007 from Proposed → Accepted by *shipping* the n=300 fixture corpus + executing the ci-full run + landing the v2.0.0 manifest cells. Cite arxiv:2503.01747 in the ADR for the n=300 inflection-point rationale. Out of scope: corpus synthesis methodology (already in ADR-0005); pre-staging is already in ADR-0007.

5. **Floor the MCP SDK pin at `^1.27.1` in `package.json` + `supabase/functions/mcp/deno.json`.** Captures the `onerror` fix (PR #1580, merged 2026-02-24). Don't pin to a specific minor — the SemVer 2.0.0 contract makes additive minors safe. Documentation in `references/edge-deployment.md` § "Architecture choice" gets a one-line note about the floor rationale. **Mechanical change.**

6. **Frame the ADR as "v1.0.0 ship surface" — not "completeness."** "Completeness" implies "we believe everything is covered"; the artifact will never be complete in that sense (Presence stays deferred per ADR-0004; manifest n=1,000 stays a v3 question; multi-tenant fan-out stays a v2 hardening question). "Ship surface" is the right framing — *what we commit on for v1.0.x consumers*.

## Falsifiable predicted effect (draft)

> **An external consumer of `supabase-realtime-skill@1.0.0` who runs `supabase functions deploy --no-verify-jwt mcp` against a Pro project, then exercises JSON-RPC `tools/call` for all five tools (watch_table, broadcast_to_channel, subscribe_to_channel, list_channels, describe_table_changes) on the deployed URL with a forwarded user JWT, observes:**
> - **All five tool routes return non-error responses (or appropriate structured error envelopes for invalid inputs)**
> - **`watch_table` delivers at least one Postgres-Changes event within a 30s wall budget against a freshly-published table**
> - **`subscribe_to_channel` receives a fan-out from a separate `broadcast_to_channel` invocation within a 10s wall budget**
> - **The ci-full run at n=300 PASSes the v2.0.0 manifest gates including the previously-unreachable CI-upper cells**
> - **The deployed function logs include `onerror` callbacks for transport-layer failures (rather than swallowing them silently)**

Properties:
- **Smoke-test-shaped + manifest-shaped.** Same dual-receipt pattern as ADR-0014 (which had the multi-tenant smoke on real Pro branches AND the manifest gate at n=100). v1.0.0 extends this to *full surface coverage*.
- **Falsifiable in seven named directions:**
  - If `watch_table` smoke fails because the warm-up window is shorter than +5s on Edge, the multi-insert schedule needs revision (or the Edge isolate cold-start absorbs the warm-up — possible but unverified).
  - If `subscribe_to_channel` fails to receive the fan-out, either the listener client hit the dedup foot-gun, or RLS on a public channel is denying (ADR-0011/0013 covered private; public should be open).
  - If the n=300 corpus synthesis introduces label drift past ADR-0005's 5% audit threshold, the corpus needs re-curation before the v2.0.0 manifest gates can be trusted.
  - If the v2.0.0 manifest's CI-upper cells fail at n=300 (i.e., the substrate has a higher base rate than v1.x assumed), the threshold values need re-justification — that's a substrate-correctness conversation, not a methodology one.
  - If the `^1.27.1` SDK floor breaks the build (transitive peer-dep conflict), the floor needs adjustment — low risk per external research, but smoke is the gate.
  - If the Edge isolate's 150s ceiling composes badly with the new smokes' wall budget (30s + 10s), the smoke order needs serializing — but each `it()` block runs sequentially in vitest by default, so this is theoretical.
  - If `gh api` checks in subsequent recons surface that #2029 *did* land between filing and shipping, the multi-insert pattern can simplify — but the recon doesn't depend on this.
- **Cost:** ~$30-60 for n=300 corpus synthesis + 2-3 ci-full runs at $6-9 each + ~$0 for the smoke extensions (re-uses the existing host project + fetchProjectKeys helpers). Total: ~$50-80 for the v1.0.0 evidence package.

## Where design risk concentrates

1. **The `watch_table` Edge smoke is unmapped — first deploy + smoke is the actual proof.** No public prior art. The internal precedent (`tests/smoke/watch-table.smoke.test.ts`) covers npm-package-against-real-branch; the Edge variant exercises the same primitive via JSON-RPC over HTTPS, but the warm-up window timing on Edge isolates is not characterized. Risk: warm-up may be *shorter* on Edge (cold-start absorbs it) — in which case the multi-insert schedule still works but is over-engineered. Risk: warm-up may be *longer* on Edge if isolate startup serializes against websocket connect — in which case +5s / +10s isn't enough.

2. **n=300 corpus synthesis is the heaviest lift in the bundle.** ADR-0005's data-quality audit playbook applies (hand-curated seeds, LLM-augmented at controlled ratios, spot-checked at 5% sample). The 200 net-new fixtures past the existing n=100 are a separate engineering pass. If the corpus synthesizer produces drift past audit threshold, the v2.0.0 manifest can't ship — the v1.0.0 ship event blocks on this.

3. **Manifest v2.0.0 vs npm v1.0.0 are independently versioned (per ADR-0001 + ADR-0007).** v1.0.0 npm release ships with manifest v2.0.0; the version-stream split needs a clear note in the README and CHANGELOG so consumers don't conflate them. (CLAUDE.md already lampshades this in the ADR-0007 reference.)

4. **The `subscribe_to_channel` smoke's "separate sender" pattern is RLS-public only.** Multi-tenant private-channel fan-out smoke would require the full RLS-policy provisioning we already validated in `tests/smoke/multi-tenant-rls.smoke.test.ts`. v1.0.0 ships the public-channel smoke; the private-channel smoke on Edge stays a v2 hardening item. Document this in the ADR as deliberate — the multi-tenant primitive is verified at the npm-package layer (ADR-0011, ADR-0013); Edge re-verification adds rigor but not new substrate-correctness coverage.

5. **`^1.27.1` floor advertises the `onerror` fix but doesn't *exercise* it.** If we wanted the v1.0.0 smoke to prove the `onerror` callback fires, we'd need a deliberate transport-error scenario. Out of scope; the floor is mechanical and the prior-art change is well-documented at upstream — calling that "covered" is reasonable for v1.0.

6. **CLAUDE.md currency drift accumulates.** Once v1.0.0 ships, the same lines that drifted after ADR-0015 (52, 56, 175, 177) drift again. The CLAUDE.md sweep-PR pattern (mirror of `b5078a0` + `043cc75`) needs to be a habit, not a one-off.

7. **T31 (the upstream `supabase/agent-skills` submission) stays open.** ADR-0004 already framed it as "user feedback shape, not as-drafted." v1.0.0 ships independently; T31 is a parallel external surface decision that doesn't gate the npm release.

## What this means for the next step

**Direction:** narrow v1.0.0 ship — three smoke extensions + manifest v2.0.0 + SDK floor. Same evidence-shape as the prior recons. NOT a substrate-correctness ADR (no broken behavior). NOT a worked-example ship (the worked examples are correct). Closes the substrate-completeness gap that ADR-0015 left open at the surface level.

**Recommended ADR pre-loads:**

- **Sequence smoke extensions before manifest v2.0.0.** The smokes are mechanical (re-use existing helpers); the manifest is a corpus-synthesis lift. If the corpus expansion blocks, the smokes still ship as a `v0.3.0` interim release that closes the verification gap without claiming v1.0 status.
- **`watch_table` smoke first** (highest novelty, biggest risk), then `subscribe_to_channel` (easier; no warm-up complications), then SDK floor bump (mechanical).
- **Manifest v2.0.0 as a separate PR.** Ship it last — the smoke PR is the substrate completeness; the manifest is the methodology completeness. They compose for v1.0 but don't depend on each other.
- **Lead the v1.0.0 ADR with the falsifiable predicted effect**, then list the six decisions, then trace the receipts back to each. Same structure as ADR-0015.
- **Frame v1.0.0 in the README as "what consumers can expect from 1.x"** — no API changes for at least one minor; manifest gates honored at n=300; Edge deploy verified end-to-end across all five tools. Short statement, not a full contract doc.

These are recommendations, not decisions — ADR will be filed as **Proposed**, per ADR status discipline.

**Open questions deferred to the ADR pass:**

- Whether v1.0.0 bundles all four sub-items (3 smokes + manifest) into one PR or splits into two. Tilt: split — smoke PR is mechanical and easy to review; manifest PR has a corpus-synthesis blob worth reviewing separately.
- Whether the `watch_table` Edge smoke uses `support_tickets` (host project schema dependency, already working post-eval-runs) or creates a temp table inline. Tilt: inline temp table — same self-containment principle as the ADR-0015 describe smoke (which uses a `Date.now()`-suffixed ghost table).
- Whether the `subscribe_to_channel` Edge smoke times out at 10s (matching Supabase's own integration test wall-time) or 30s (matching our existing watch_table.smoke). Tilt: 10s — it's a fan-out, not a warm-up-bound subscription; broadcast doesn't have the postgres_changes warm-up window.
- Whether to also extend the Edge smoke for `list_channels` after a broadcast (to verify registry mutation propagates via the per-request stateless server). Probably not — `list_channels` returns empty across requests because the server is per-request stateless; testing that empty-across-requests is a feature-not-bug is more confusing than valuable.
- Whether the v1.0.0 ADR should also commit on a `CHANGELOG.md` discipline going forward (one section per minor; one PR per minor). Tilt: yes, but as a single sentence — discipline is light-touch.
- Whether to call the next ADR `0016` or a "v1.0.0 release manifest" composite ADR. The ADR-0015 pattern was one-ADR-per-coherent-decision-bundle; this fits — `0016-v1.0.0-ship-surface.md`.
