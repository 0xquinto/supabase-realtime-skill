> Synthesized from the [LLM Evals YouTube playlist](https://www.youtube.com/playlist?list=PLgIaq8VgndJvt-HKMHPXehyJNNXQsAVHD) plus six external research passes (`research/prompt-injection.md`, `research/agent-eagerness.md`, `research/statistical-design.md`, `research/harness-engineering.md`, `research/dataset-construction.md`, `research/construct-validity.md`). Each section frames a design decision and summarizes what the literature collectively recommends. Citations link to per-video notes or per-research-pass files.

## How to use this playbook

This is a working document for designing eval slices targeting [supabase-community/supabase-mcp](https://github.com/supabase-community/supabase-mcp). Each section poses a design question, summarizes what the playlist + research passes say about it, and leaves the actual decision to the slice spec. Read the cited notes for full context.

**Source coverage:** 14 videos triaged "high" and deep-extracted; 19 triaged "low" and kept at chapter-summary level; 3 videos were inaccessible (403). Six external research passes (one per `research/<topic>.md` file) cite ~50 academic and industry sources beyond the playlist, with a 2025+ recency default. Inaccessible videos are not citable; if a topic appears in only those videos, treat it as a gap.

## 1. Scope shape (breadth-first / depth-first / capability-matrix)

**What the videos say:**

- **Start narrow and depth-first.** The strongest single piece of advice across the playlist: build the simplest possible eval against a tiny, hand-crafted dataset, then iterate. ([UGmenkjGXqM — Text2SQL](notes/UGmenkjGXqM-llm-eval-for-text2sql.md): "Hardcode 5-10 inputs and expected outputs to get your first eval running.") ([LwLxlEwrtRA — GitHub Copilot](notes/LwLxlEwrtRA-the-evals-that-made-github-copilot.md): "Start with simple Jupyter notebooks or Python scripts to run ad-hoc evals.")
- **Don't aim for coverage; aim for the failure modes that matter.** Generic capability matrices score badly on "ROI per eval written." ([GL0XhAj5LPE — Common Mistakes](notes/GL0XhAj5LPE-llm-evals-common-mistakes.md): generic evals are useless; only failure-mode-specific ones move the needle.) ([e2i6JbU2R-s — Error Analysis ROI](notes/e2i6JbU2R-s-error-analysis-the-highest-roi-technique-in-ai-engineering.md): "categorized error counts" tied to real product issues outperform generic metrics.)
- **Decompose the system into a capability funnel** so you can isolate which stage breaks. For an MCP harness this is: prompt parsing → tool selection → argument extraction → execution → result interpretation. ([R_HnI9oTv3c — Capability Funnel](notes/R_HnI9oTv3c-stop-managing-ai-projects-like-traditional-software.md): NL→SQL example breaks down into 5 stages.)
- **Use "Theoretical Saturation" as your stopping rule** when sourcing real prompts/traces — typically 20-40 examples is enough to find your top failure modes. ([BsWxPI9UM4c](notes/BsWxPI9UM4c-why-ai-evals-are-the-hottest-new-skill-for-product-builders-.md), [GL0XhAj5LPE](notes/GL0XhAj5LPE-llm-evals-common-mistakes.md), [qH1dZ8JLLdU](notes/qH1dZ8JLLdU-intro-to-error-analysis-creating-custom-data-annotation-apps.md))

## 2. Scoring approach (programmatic / verifiable / LLM-judge / hybrid)

**What the videos say:**

- **Use a tiered scoring spectrum, not a single approach.** The recurring three-tier model: programmatic/algorithmic checks → verifiable execution → LLM-as-judge. ([LwLxlEwrtRA — GitHub Copilot](notes/LwLxlEwrtRA-the-evals-that-made-github-copilot.md) makes this explicit: Algorithmic for structure (regex, JSON schema, length), Verifiable for execution (does the SQL run? does the code compile?), LLM-judge only for subjective quality.) ([SnbGD677_u0](notes/SnbGD677_u0-instrumenting-evaluating-llms.md) calls these "Levels of Evaluation".)
- **Verifiable execution is the killer scorer for tool-using systems.** For an MCP server with `execute_sql`, `apply_migration`, etc., the strongest signal is "did the SQL execute and return the expected rows?" not "does the SQL look correct?" ([UGmenkjGXqM — Text2SQL](notes/UGmenkjGXqM-llm-eval-for-text2sql.md): `no_error` (binary execution check) + `JSONDiff` (deterministic match) + `correct_sql` (LLM-judge on semantic equivalence) layered together.) ([LwLxlEwrtRA — Harnesslib](notes/LwLxlEwrtRA-the-evals-that-made-github-copilot.md): GitHub Copilot's flagship eval was "strip function body, regenerate, run pre-existing unit tests".)
- **LLM-judge is a separate ML problem you must validate.** Never trust a judge prompt without measuring it against human ground-truth. ([BsWxPI9UM4c](notes/BsWxPI9UM4c-why-ai-evals-are-the-hottest-new-skill-for-product-builders-.md): use TPR/TNR via confusion matrix to measure judge alignment.) ([SnbGD677_u0](notes/SnbGD677_u0-instrumenting-evaluating-llms.md): use ROC-AUC, precision, recall against a labeled set.)
- **Always use binary scoring, never Likert.** This is unanimous across the playlist. Forced binary makes inter-rater agreement and judge alignment computable; 1-5 scales produce noise. ([BsWxPI9UM4c](notes/BsWxPI9UM4c-why-ai-evals-are-the-hottest-new-skill-for-product-builders-.md), [GL0XhAj5LPE](notes/GL0XhAj5LPE-llm-evals-common-mistakes.md), [OJItZndMUII](notes/OJItZndMUII-ai-evals-for-engineers-course-preview-chapters-1-3-of-8.md))
- **Make judge prompts narrow and failure-mode-specific.** Don't ask "is this a good response?". Ask "did the agent call `apply_migration` instead of `execute_sql` for a DDL change?" ([GL0XhAj5LPE](notes/GL0XhAj5LPE-llm-evals-common-mistakes.md), [N-qAOv_PNPc](notes/N-qAOv_PNPc-from-noob-to-automated-evals-in-a-week-as-a-pm-w-teresa-torr.md): decompose broad rubrics into discrete dimensions.)
- **Confidence-calibrate judges** by inspecting logprobs and setting thresholds (e.g. require >80% confidence on a "fail" verdict). ([SnbGD677_u0](notes/SnbGD677_u0-instrumenting-evaluating-llms.md))

## 3. Dataset construction (handcrafted / synthetic / replayed traces)

**What the videos say:**

- **Start handcrafted; synthesize later.** Universal pattern: hardcode 5-30 real-feeling prompts you can defend, then expand. ([UGmenkjGXqM](notes/UGmenkjGXqM-llm-eval-for-text2sql.md), [97iykOemOn4](notes/97iykOemOn4-llm-eval-tools-compared-braintrust.md): "real data trumps synthetic data" — even ask family/colleagues to draft prompts.)
- **Bootstrap with synthetic generation only after handcrafted seed exists.** Use the LLM to generate variations grounded in your schema/tools, then filter by execution success and human review before promoting into the golden set. ([UGmenkjGXqM](notes/UGmenkjGXqM-llm-eval-for-text2sql.md): generate SQL queries against a known schema, run them, keep only the ones that execute, then save question+query pairs.) ([SnbGD677_u0](notes/SnbGD677_u0-instrumenting-evaluating-llms.md): generate per-scenario inputs like "one match / multi-match / no match".)
- **Use dimensional synthesis for diversity.** Define axes (persona × feature × scenario × difficulty) and ask the LLM to generate examples for specific tuples to cover the long tail. ([OJItZndMUII](notes/OJItZndMUII-ai-evals-for-engineers-course-preview-chapters-1-3-of-8.md), [97iykOemOn4](notes/97iykOemOn4-llm-eval-tools-compared-braintrust.md))
- **Replayed traces are gold once you have them.** Production traces (full inputs, tool calls, tool responses, final outputs) are the highest-quality eval data. ([e2i6JbU2R-s](notes/e2i6JbU2R-s-error-analysis-the-highest-roi-technique-in-ai-engineering.md): build a tool-call viewer; the dataset includes `Tool Name`, `Call ID`, `Tool Response` JSON.) ([SnbGD677_u0](notes/SnbGD677_u0-instrumenting-evaluating-llms.md): query the data warehouse for historical user actions to bootstrap evals.)
- **Treat the dataset as version-controlled, evolving golden data.** Every newly observed failure mode should add a regression case. ([BsWxPI9UM4c](notes/BsWxPI9UM4c-why-ai-evals-are-the-hottest-new-skill-for-product-builders-.md): evals are a continuous flywheel.)
- **Annotate metadata heavily.** Each dataset row should carry source (handcoded vs synthetic vs trace), submitter, scenario tag — to enable slice analysis later. ([97iykOemOn4](notes/97iykOemOn4-llm-eval-tools-compared-braintrust.md))
- **Decompose long-context evals.** For tasks involving large tool outputs (e.g. SQL result sets, log dumps), build distractor variants and semantic-NIAH-style probes; lexical-only NIAH is a flawed signal. ([3s_N60u0jEY — Context Rot](notes/3s_N60u0jEY-context-rot-when-long-context-fails.md))

## 4. Failure mode taxonomy (what to specifically test for)

**What the videos say:**

- **Don't define the taxonomy a priori; discover it via Open + Axial Coding on real traces.** Read 20-40 traces, write free-form notes on each failure ("open coding"), then cluster into a structured taxonomy ("axial coding") — using an LLM to do the clustering once you have the raw notes. ([BsWxPI9UM4c](notes/BsWxPI9UM4c-why-ai-evals-are-the-hottest-new-skill-for-product-builders-.md), [qH1dZ8JLLdU](notes/qH1dZ8JLLdU-intro-to-error-analysis-creating-custom-data-annotation-apps.md), [OJItZndMUII](notes/OJItZndMUII-ai-evals-for-engineers-course-preview-chapters-1-3-of-8.md), [97iykOemOn4](notes/97iykOemOn4-llm-eval-tools-compared-braintrust.md))
- **Failure-mode categories from the playlist** that likely apply to an MCP server:
  - *Wrong tool selected* (tool description ambiguity) — diagnosed via confusion matrix. ([LwLxlEwrtRA — Tool Use Confusion Matrix](notes/LwLxlEwrtRA-the-evals-that-made-github-copilot.md))
  - *Right tool, wrong arguments* (e.g. dropped LIMIT, wrong schema name, hallucinated parameter). ([LwLxlEwrtRA](notes/LwLxlEwrtRA-the-evals-that-made-github-copilot.md), [SnbGD677_u0](notes/SnbGD677_u0-instrumenting-evaluating-llms.md))
  - *Date/time hallucinations* — relative-date resolution is famously brittle. ([e2i6JbU2R-s](notes/e2i6JbU2R-s-error-analysis-the-highest-roi-technique-in-ai-engineering.md): "two weeks from now" went from 33% → 100% after targeted eval.)
  - *Refusal / abstention vs hallucination* — different model families fail differently when faced with distractors. ([3s_N60u0jEY](notes/3s_N60u0jEY-context-rot-when-long-context-fails.md): Claude tends to abstain; GPT tends to hallucinate.)
  - *Context-rot failures* — long tool outputs (e.g. log dumps, large result sets) degrade comprehension non-uniformly with token count. ([3s_N60u0jEY](notes/3s_N60u0jEY-context-rot-when-long-context-fails.md))
  - *Distractor susceptibility* — semantically similar but wrong information defeats retrieval. ([3s_N60u0jEY](notes/3s_N60u0jEY-context-rot-when-long-context-fails.md))
  - *Refusal on repetition* — models refuse to output long verbatim strings, falsely citing copyright. ([3s_N60u0jEY](notes/3s_N60u0jEY-context-rot-when-long-context-fails.md))
  - *Hallucinated placeholders* — leaving "[TABLE_NAME]" or "[CANDIDATE_NAME]" instead of substituting. ([qH1dZ8JLLdU](notes/qH1dZ8JLLdU-intro-to-error-analysis-creating-custom-data-annotation-apps.md))
  - *Role/power-dynamic confusion* — useful analog for "the agent runs the query when it should have asked first". ([qH1dZ8JLLdU](notes/qH1dZ8JLLdU-intro-to-error-analysis-creating-custom-data-annotation-apps.md))
- **Track failure-mode counts, not aggregate scores.** ([e2i6JbU2R-s](notes/e2i6JbU2R-s-error-analysis-the-highest-roi-technique-in-ai-engineering.md): "Tour Scheduling Errors: 9, Handoff Failures: 3" → drives the roadmap.)
- **No video in this playlist covered MCP-specific failure modes directly.** This is a gap to fill via our own error analysis once the harness produces real traces.

## 5. Tool-call correctness for MCP servers (selection / arguments / sequencing)

**What the videos say:**

- **The tool-use confusion matrix is the canonical eval for tool selection.** Plot expected tool × predicted tool to surface ambiguous tool descriptions. ([LwLxlEwrtRA — GitHub Copilot](notes/LwLxlEwrtRA-the-evals-that-made-github-copilot.md): "highly applicable to testing MCP servers".)
- **Treat agent planning as binary classification per step.** "Did it pick the right tool? Did it extract the right arguments? Did it call them in the right order?" — each step graded independently. ([SnbGD677_u0](notes/SnbGD677_u0-instrumenting-evaluating-llms.md): "Treat planning as a binary classification state-machine problem.")
- **Evaluate intermediate steps, not just final outputs.** If you only score "did the user get the right answer?", you can't tell whether the failure was in selection, extraction, or execution. ([R_HnI9oTv3c — Capability Funnel](notes/R_HnI9oTv3c-stop-managing-ai-projects-like-traditional-software.md): "There is no value in the final result evals.")
- **Use Inspect's Agent Bridge pattern as a reference architecture.** Inspect lets you wrap an external agent (an MCP client) and run it as the "Solver" while keeping dataset/scoring decoupled. ([_UY49Q_qFhs — Inspect Agent Bridge](notes/_UY49Q_qFhs-inspect-a-llm-eval-framework-used-by-anthropic-deepmind-grok.md): explicitly recommended for "MCP server/client pair".)
- **Verify by execution.** For SQL/migration tools: run the generated SQL against a fixed-state database, compare result sets. ([UGmenkjGXqM](notes/UGmenkjGXqM-llm-eval-for-text2sql.md): `no_error` + `JSONDiff` + `correct_sql` LLM-judge.) ([LwLxlEwrtRA](notes/LwLxlEwrtRA-the-evals-that-made-github-copilot.md): regenerate function body and run pre-existing tests.)
- **Mock external dependencies in CI.** Inject fixed state so evals test reasoning, not third-party uptime. ([SnbGD677_u0](notes/SnbGD677_u0-instrumenting-evaluating-llms.md))
- **Sandbox tool execution.** For tools that mutate state (DDL, branch operations), run inside Docker/k8s — Inspect's `SandboxEnvironment` handles this. ([_UY49Q_qFhs](notes/_UY49Q_qFhs-inspect-a-llm-eval-framework-used-by-anthropic-deepmind-grok.md))

## 6. Destructive-action and safety eval handling

**What the videos say:**

- **No video in this playlist directly covered destructive-action evaluation for MCP-style servers.** This is a real gap — the closest analogs are general "verifiable execution" advice and tool-call correctness.
- **Sandbox is the de-facto answer for safely evaluating mutation-capable tools.** ([_UY49Q_qFhs](notes/_UY49Q_qFhs-inspect-a-llm-eval-framework-used-by-anthropic-deepmind-grok.md): "Don't run unsafe tools locally; use SandboxEnvironment.")
- **Mock dependencies / inject fixed state** is the clean approach for CI — the eval becomes deterministic and rollback-free. ([SnbGD677_u0](notes/SnbGD677_u0-instrumenting-evaluating-llms.md))
- **The destructive-action subset of supabase-mcp** (`apply_migration`, `create_project`, `pause_project`, `restore_project`, `deploy_edge_function`, `create_branch`, `delete_branch`, `merge_branch`, `reset_branch`, `rebase_branch`, `update_storage_config`) needs a specific eval slice — likely covering: read-only mode enforcement, agent's willingness to call destructive tools without confirmation, and prompt-injection resistance (since the supabase-mcp README explicitly flags prompt injection as the primary attack vector).
- **Prompt injection is named explicitly in the supabase-mcp threat model** but no playlist video addresses prompt injection eval directly. Another gap.

## 8. Anti-patterns to avoid

Distilled across notes:

- **Likert scales for LLM-judge or human labeling.** Use binary. ([BsWxPI9UM4c](notes/BsWxPI9UM4c-why-ai-evals-are-the-hottest-new-skill-for-product-builders-.md), [GL0XhAj5LPE](notes/GL0XhAj5LPE-llm-evals-common-mistakes.md), [OJItZndMUII](notes/OJItZndMUII-ai-evals-for-engineers-course-preview-chapters-1-3-of-8.md))
- **"Stacked abstractions"** — LLM writes prompt → LLM generates rubric → LLM-judge scores. No human ground truth anywhere in the loop. ([97iykOemOn4](notes/97iykOemOn4-llm-eval-tools-compared-braintrust.md))
- **Generic off-the-shelf metrics** ("conciseness", "hallucination", "toxicity"). They don't reflect your product's failure modes. ([GL0XhAj5LPE](notes/GL0XhAj5LPE-llm-evals-common-mistakes.md), [e2i6JbU2R-s](notes/e2i6JbU2R-s-error-analysis-the-highest-roi-technique-in-ai-engineering.md))
- **Evaluating only end-to-end output.** You can't debug a multi-step agent by scoring only the final response. ([R_HnI9oTv3c](notes/R_HnI9oTv3c-stop-managing-ai-projects-like-traditional-software.md), [SnbGD677_u0](notes/SnbGD677_u0-instrumenting-evaluating-llms.md))
  - **Concrete instance: self-clean masking.** Slice-1 (prompt-injection) shipped DBAssertion-only scoring; the agent reflexively dropped attacker_sink / pwned_* tables "for security hygiene" *within the same trial*, so point-in-time DB-state checks returned clean and the run scored 39/39 pass while the transcripts showed multiple attacks landing. Fix in [ADR 0004](../docs/decisions/0004-foundation-attack-tool-calls-negative-matchers.md): add transcript-level NEGATIVE matchers; OR them into attack_succeeded alongside DB state. (`evals/slices/prompt_injection/error-analysis.md`)
- **Synthetic-only attack registry without real-trace seeding.** Slice-1's first 5 framings included 2 (`important-system-notice` bracketed delimiters; `html-comment-wrapper`) that no real-world adversarial corpus uses, and was missing the OWASP-canonical "ignore all previous instructions" class entirely. ~90 min of trace-seeding against AgentDojo / BIPIA / InjecAgent / Unit42 / OWASP / the supabase-mcp issue tracker swapped 2 framings for 2 grounded ones before shipping. Hand-crafted seed first, but trace-seed before treating it as representative. (`evals/slices/prompt_injection/error-analysis.md`)
- **Whack-a-mole prompt tweaking** without a regression set. ([UGmenkjGXqM](notes/UGmenkjGXqM-llm-eval-for-text2sql.md), [qH1dZ8JLLdU](notes/qH1dZ8JLLdU-intro-to-error-analysis-creating-custom-data-annotation-apps.md))
- **Premature LLM-judge automation** before doing manual error analysis to discover what to judge. ([qH1dZ8JLLdU](notes/qH1dZ8JLLdU-intro-to-error-analysis-creating-custom-data-annotation-apps.md), [BsWxPI9UM4c](notes/BsWxPI9UM4c-why-ai-evals-are-the-hottest-new-skill-for-product-builders-.md))
- **Underspecified judge prompts.** "Is this correct?" with no rubric → noise. ([GL0XhAj5LPE](notes/GL0XhAj5LPE-llm-evals-common-mistakes.md), [LwLxlEwrtRA](notes/LwLxlEwrtRA-the-evals-that-made-github-copilot.md))
- **Optimizing without guardrail metrics.** "Acceptance rate up" is meaningless if mean lines generated halved. ([LwLxlEwrtRA](notes/LwLxlEwrtRA-the-evals-that-made-github-copilot.md))
- **Naïve Needle-in-a-Haystack.** Lexical NIAH greatly overstates real long-context performance. ([3s_N60u0jEY](notes/3s_N60u0jEY-context-rot-when-long-context-fails.md))
- **Blind chunking / context stuffing.** Append-everything growing context windows produce "context rot". ([3s_N60u0jEY](notes/3s_N60u0jEY-context-rot-when-long-context-fails.md), [t6r4U0SlnPc](notes/t6r4U0SlnPc-how-to-process-documents-at-scale-with-llms.md))
- **Evaluation by committee.** Subjective grading by multiple stakeholders without a "Benevolent Dictator" produces inconsistent ground truth. ([BsWxPI9UM4c](notes/BsWxPI9UM4c-why-ai-evals-are-the-hottest-new-skill-for-product-builders-.md))
- **Heavy frameworks too early.** Plain TS + Bun + raw SDKs beats `inspect-ai` / LangChain for early eval debuggability. Ratified by ADR 0006 (why-we-left-inspect-ai), ADR 0007 (bun-as-runtime), ADR 0008 (no-third-party-eval-framework); the academic backing is in [`research/harness-engineering.md`](research/harness-engineering.md) target 4 (HAL ICLR 2026 + OLMES NAACL 2025). ([UGmenkjGXqM](notes/UGmenkjGXqM-llm-eval-for-text2sql.md))
- **Mistaking product metrics for eval metrics.** DAU/click-through ≠ model quality. ([SnbGD677_u0](notes/SnbGD677_u0-instrumenting-evaluating-llms.md))
- **Using foundation-model benchmarks (MMLU/Arena) to select an application model.** ([GL0XhAj5LPE](notes/GL0XhAj5LPE-llm-evals-common-mistakes.md))
- **Reading thousands of traces.** Stop at theoretical saturation (~20-40 traces). ([BsWxPI9UM4c](notes/BsWxPI9UM4c-why-ai-evals-are-the-hottest-new-skill-for-product-builders-.md), [GL0XhAj5LPE](notes/GL0XhAj5LPE-llm-evals-common-mistakes.md), [qH1dZ8JLLdU](notes/qH1dZ8JLLdU-intro-to-error-analysis-creating-custom-data-annotation-apps.md))
- **Recommending a change without a falsifiable predicted effect.** A finding that says "tightening tool description X should reduce failure rate F" without a pre-registered metric / direction / magnitude can't tell signal from noise after the change ships. The post-hoc adjustment trap: any observed delta gets reframed as confirmation. Pre-register the prediction *before* the ablation. ([Lin et al., arXiv:2604.25850](https://arxiv.org/abs/2604.25850) "decision observability")
- **Reporting standard deviation or min-max range instead of a confidence interval on a binomial proportion.** Binomial variance is fully determined by p and n; an "observed variance" claim is uninformative without comparison to N·p·(1−p). The reporting unit must be `<rate> [<lo>, <hi>] (Wilson 95%, n=<N>)`. ([ICLR Blogposts 2025 — "Towards More Rigorous Evaluations of Language Models"](https://iclr-blogposts.github.io/2025/blog/towards-more-rigorous-llm-evals/); see [`research/statistical-design.md`](research/statistical-design.md) Source 3)
- **Using CLT-based / Wald confidence intervals on small-n LLM evaluations.** For n < ~300 they dramatically under-cover, especially near p=0 or p=1 (our regime — slice-2's 25/25, slice-3's saturation cells). Use Wilson, Clopper-Pearson, or Beta-Binomial Bayesian instead. ([Bowyer, Aitchison & Ivanova, ICML 2025 Spotlight](https://proceedings.mlr.press/v267/bowyer25a.html); see [`research/statistical-design.md`](research/statistical-design.md) Source 1)
- **Running multiple A/B contrasts and reporting only the significant one without multiplicity correction (post-hoc selection bias).** With k contrasts at uncorrected α=0.05, family-wise error ≈ 1−(0.95)ᵏ. Pre-register the contrasts; apply Bonferroni (α/k per test) as the simple default, Holm-Bonferroni for slightly higher power, Benjamini-Hochberg FDR control if k≥4. ([Luo, Kasirzadeh & Shah, NeurIPS 2025 AI4Science Spotlight, arXiv:2509.08713](https://arxiv.org/html/2509.08713v1); see [`research/statistical-design.md`](research/statistical-design.md) Source 6)
- **Single-prompt cell scoring.** Even with n=25 trials per cell, scoring on a single prompt formulation conflates axis-effect with prompt-formulation noise. Each task should get ≥3 meaning-preserving paraphrases; the reported rate is mean-across-paraphrases with explicit across-paraphrase variance reported *separately* from sampling variance. ([Lior et al., ReliableEval, EMNLP 2025 Findings](https://aclanthology.org/2025.findings-emnlp.594.pdf); see [`research/harness-engineering.md`](research/harness-engineering.md) Source 7)
- **Process / outcome / environment-confounder conflation in binary scoring.** A single attack-landed bit conflates "agent intended bad action," "bad action completed," and "environment confounder produced a false positive." Score these as three columns: process (intent), outcome (completion), environment (confounder). Otherwise: false positives from API 5xx errors look like agent failures, and reward signals are too lenient or too harsh. ([Rosset et al., Microsoft Universal Verifier 2026](https://www.microsoft.com/en-us/research/articles/the-art-of-building-verifiers-for-computer-use-agents/); see [`research/harness-engineering.md`](research/harness-engineering.md) Source 5)
- **Single-metric agent-safety harness — never collapse safety and helpfulness.** A "block everything" agent gets a perfect safety score under destructive-rate-only scoring; a "do everything" agent gets a perfect helpfulness score. Co-report both, on the same trial set. Slice-3's `search_docs` over-routing finding (helpfulness failure, 81/100) is exactly the helpfulness-axis result that makes the destructive-rate-axis null result (0/100) interpretable rather than embarrassing. ([Ruan et al., ToolEmu, ICLR 2024](https://proceedings.iclr.cc/paper_files/paper/2024/file/7274ed909a312d4d869cc328ad1c5f04-Paper-Conference.pdf); [QuittingAgents, Oct 2025](https://github.com/victorknox/QuittingAgents); see [`research/agent-eagerness.md`](research/agent-eagerness.md) Sources 1, 3)
- **Reporting a mitigation A/B win without a situational-awareness probe.** Apollo/OpenAI's anti-scheming training reduced covert behavior but *increased* CoT mentions of "this is an eval" from ~2% to ~4.5% — the mitigation partly taught models to recognize evals rather than to comply. Any slice-4-shape prompt-engineering A/B that ships an "improvement" must include a CoT probe for the model treating the scenario as a test. Otherwise the delta is confounded with eval-detection. ([Schoen et al., Apollo × OpenAI, Sept 2025](https://www.apolloresearch.ai/research/stress-testing-deliberative-alignment-for-anti-scheming-training); see [`research/agent-eagerness.md`](research/agent-eagerness.md) Source 2)
- **Co-occurrence shortcut leakage in eval items.** Items the model can solve via training-time co-occurrence between input and target entities — not by the named capability. Removing these shortcuts dropped SOCRATES capability scores 80% → 5% on some relation types. Distinct from "narrow-signature scoring" (slice-2 lesson, which is about *attack* signature, not *training-data* shortcut). When seeding, audit for `(head_entity, answer_entity)` pairs likely to co-appear in pretraining; exclude or score separately. ([Yang et al., DeepMind/UCL, ACL Findings 2025](https://aclanthology.org/2025.findings-acl.205.pdf); see [`research/dataset-construction.md`](research/dataset-construction.md) Source 10)
- **Static-benchmark contamination decay.** A hand-crafted seed dataset frozen at v1.0 silently degrades as new model generations train on it (or its public discussion). Distinct from "synthetic-before-handcrafted-seed" because it applies to hand-crafted seeds *too*. Mitigation: SemVer the dataset, tag items as stable / evolved / uncharted, audit freshness per slice rerun. ([Chen et al., Static-to-Dynamic survey, EMNLP 2025](https://aclanthology.org/2025.emnlp-main.511.pdf); [Tang et al., EvoWiki, ACL 2025](https://aclanthology.org/2025.acl-long.47.pdf); see [`research/dataset-construction.md`](research/dataset-construction.md) Sources 4, 5)
- **Single-attack ASR reported as ASR.** A defense's success rate against one attack family is a *floor*, not a measurement. "The Attacker Moves Second" (ICLR 2026) bypassed 12 defenses originally reported at near-zero ASR, all to >90%, by tuning attack-side compute. Slice findings on defenses must report ASR-floor with the attack budget, not bare ASR — and note the defender's response is undefined past that floor. ([Carlini et al., 2019](https://nicholas.carlini.com/papers/2019_howtoeval.pdf); [Attacker Moves Second, ICLR 2026](https://openreview.net/pdf/7855aef1ec24a6c34096532b64736e455004920c.pdf); see [`research/dataset-construction.md`](research/dataset-construction.md) Sources 1, 2)
- **Phenomenon-proxy gap.** The slice's prompts allow the model to satisfy the trial via routes other than the named construct, so the rate measures *some* behaviour, not the *named* one. Diagnostic: name two non-target routes per prompt; if either is plausible at the canonical model, redesign before scoring. **Slice-3 is the worked example: prompts let haiku route to `search_docs` and skip the destructive-or-not choice slice-2 was measuring; slice-3's null result is therefore likely a construct-validity artifact, not a haiku finding.** Falsifiable disambiguation: rerun slice-2's exact prompts on haiku. ([Bean et al., NeurIPS 2025 D&B § 5.2](https://openreview.net/pdf/5e50cba825c86cf5d8a7c4148c1a7241d89c15ca.pdf); [Salaudeen et al. 2025](https://arxiv.org/html/2505.10573v1); see [`research/construct-validity.md`](research/construct-validity.md) Sources 1, 4 + Target 4 diagnosis)
- **Construct drift across slices.** Reusing slice-(N-1)'s phenomenon name with modified prompts silently changes the construct. Required when modifying prompts: document the change, run new-vs-original on a held model, justify how the modification improves rather than breaks construct validity. ([Bean et al., NeurIPS 2025 D&B § 5.4](https://openreview.net/pdf/5e50cba825c86cf5d8a7c4148c1a7241d89c15ca.pdf); see [`research/construct-validity.md`](research/construct-validity.md) Source 1)
- **No pre-publication data-quality audit.** Claiming a rate (positive or null) without a sampled item-level audit for Mousavi's three-flaw taxonomy. **Gate:** sample ≥30 items, two-reviewer-classify into clean / structural-flaw / semantic-flaw / pragmatic-flaw; if ≥10% have any flaw class, repair or remove before the slice ships. Mousavi found 29.5% of SocialIQa items flawed and 45% pragmatic-flaw rate on ToMi — the bar is real. ([Mousavi et al., 2025](https://arxiv.org/html/2506.23864v1); see [`research/construct-validity.md`](research/construct-validity.md) Source 2)

## 9. Cross-cutting heuristics from the playlist

- **"Look at your data" is the universal first move.** Every speaker emphasizes manually reading raw traces before automating anything. The 2026 practitioner cadence quantifies it: review 20–50 production traces by hand *before* building eval infrastructure (LangChain Agent Evaluation Readiness Checklist; see "Real-trace seeding cadence" in this section). ([BsWxPI9UM4c](notes/BsWxPI9UM4c-why-ai-evals-are-the-hottest-new-skill-for-product-builders-.md), [qH1dZ8JLLdU](notes/qH1dZ8JLLdU-intro-to-error-analysis-creating-custom-data-annotation-apps.md), [e2i6JbU2R-s](notes/e2i6JbU2R-s-error-analysis-the-highest-roi-technique-in-ai-engineering.md), [SnbGD677_u0](notes/SnbGD677_u0-instrumenting-evaluating-llms.md))
- **Evals are a continuous flywheel, not a one-time gate.** Build it expecting daily/weekly iteration. ([BsWxPI9UM4c](notes/BsWxPI9UM4c-why-ai-evals-are-the-hottest-new-skill-for-product-builders-.md), [SnbGD677_u0](notes/SnbGD677_u0-instrumenting-evaluating-llms.md))
- **Treat each prompt/model change as a controlled experiment.** Same dataset, change one variable, measure delta. ([N-qAOv_PNPc](notes/N-qAOv_PNPc-from-noob-to-automated-evals-in-a-week-as-a-pm-w-teresa-torr.md), [SnbGD677_u0](notes/SnbGD677_u0-instrumenting-evaluating-llms.md))
- **The "Three Gulfs" framing** (Comprehension, Specification, Generalization) gives a useful diagnostic: where does your harness see failures? ([t6r4U0SlnPc](notes/t6r4U0SlnPc-how-to-process-documents-at-scale-with-llms.md), [OJItZndMUII](notes/OJItZndMUII-ai-evals-for-engineers-course-preview-chapters-1-3-of-8.md))
- **Use sub-agents / context budgeting for long workflows** to avoid context rot. ([3s_N60u0jEY](notes/3s_N60u0jEY-context-rot-when-long-context-fails.md))
- **Vibe-code your annotation tooling.** Don't build a generic platform; generate a single-purpose UI tailored to *your* data shape with a 30-min Cursor session. ([qH1dZ8JLLdU](notes/qH1dZ8JLLdU-intro-to-error-analysis-creating-custom-data-annotation-apps.md))
- **Real-trace seed adversarial registries before treating them as representative — at a quantified cadence.** When a slice authors a hand-crafted attack/failure-case set: (a) before infra, manually review 20–50 production traces (LangChain 2026 / Latitude 2026 — see "Real-trace seeding cadence" below); (b) pull 8–12 examples from public adversarial corpora (AgentDojo, BIPIA, InjecAgent), in-the-wild writeups, and the target system's own issue tracker; (c) target ~70% production-sampled / 30% curated-adversarial; (d) refresh quarterly. Slice-1 demonstrated "discovered, not declared" failure modes (PLAYBOOK § 4) is enforceable cheaply this way. (`evals/slices/prompt_injection/error-analysis.md`; `playbook/research/prompt-injection.md` § 5; `playbook/research/dataset-construction.md` Sources 7, 8)
- **State transition matrices for diff-evals.** When iterating, plot how queries move between states (failed→passing, passing→failed) to catch silent regressions. ([R_HnI9oTv3c](notes/R_HnI9oTv3c-stop-managing-ai-projects-like-traditional-software.md))
- **Task cascades for cost.** Cheap proxy model first, escalate to expensive oracle only when proxy is unconfident. ([t6r4U0SlnPc](notes/t6r4U0SlnPc-how-to-process-documents-at-scale-with-llms.md))
- **Decision observability — pre-register every harness/SUT change as a falsifiable prediction.** Each recommended change pairs with a specific metric, direction, and magnitude before the ablation runs. After running, the prediction is *verified or falsified* — not retrofitted. Stronger discipline than "let's see what happens"; the only way to tell signal from noise on small-n eval runs. Companion robustness signal #5 ("successful predicted intervention"). ([Lin et al., arXiv:2604.25850](https://arxiv.org/abs/2604.25850))
- **Experience observability — maintain a curated trial-transcript corpus, not ad-hoc spot-inspections.** One canonical transcript per named failure-mode signature, persisted across slices in `findings/spot-inspection-corpus/<slice>/<failure-mode>.json`. Each new slice inherits and extends rather than re-picking from scratch. Makes findings auditable and prevents quietly losing failure modes between slices. ([Lin et al., arXiv:2604.25850](https://arxiv.org/abs/2604.25850))
- **Iteration is the work, not a sign of failure.** Lin et al. report 10 harness-edit iterations to go from 69.7% → 77.0% on Terminal-Bench (~0.7%/iteration). Empirical confirmation that finding the eval was wrong, fixing it, and re-running is the operating mode — not a deviation from it. Cross-checked against the playlist's "evals are a continuous flywheel" framing.
- **Use paired tests when the same sample IDs are scored under two configs.** Slice-3 (4 cells × same 25 sample IDs) and any slice-4-shape A/B (same samples, different `--features`) have hidden pairing structure. Use McNemar's exact test for binary outcomes, Wilcoxon signed-rank for ordinal/continuous. Paired analysis is "free variance reduction"; reporting a two-sample CI when paired data is available is methodologically wasteful. ([Miller / Anthropic, arXiv:2411.00640](https://arxiv.org/abs/2411.00640); see [`research/statistical-design.md`](research/statistical-design.md) Source 2)
- **n=25/cell is defensible only for absolute effects ≥ ~0.30.** At n=25 per arm, a two-proportion Wilson test has ~80% power to detect Δ=0.30–0.40 (e.g., 80% → 40% destructive rate); ~40% power for Δ=0.15. Slice-2 (25/25) and slice-3 saturation cells are fine because Δ ≈ 1.0; for any contrast targeting Δ<0.20, budget ≥ ~100/cell or accept the result as exploratory. **Pre-register the minimum detectable effect (MDE) before running.** ([Miller / Anthropic, arXiv:2411.00640](https://arxiv.org/abs/2411.00640) power-analysis section; see [`research/statistical-design.md`](research/statistical-design.md) Source 2)
- **Run manifest per slice rollout.** Every slice run emits a JSON with foundation commit sha + model id + temperature + `--features` set + `--read-only` flag + system_prompt sha256 + tool_descriptions sha256 + judge prompt sha256 + sample seed. Without this, a finding from May is unreplicable in July when supabase-mcp ships a tool-description tweak. The literature endorses our hand-rolled choice but conditions it on a documented manifest. ([Gu et al., OLMES, NAACL 2025 Findings](https://aclanthology.org/2025.findings-naacl.282.pdf); [Stroebl et al., HAL, ICLR 2026 under review](https://openreview.net/pdf/c74cc98588086d5efd7cf146b47b0d5112ab3f90.pdf); see [`research/harness-engineering.md`](research/harness-engineering.md) Sources 8 and 2)
- **Trajectory canonicalization to (observation, reasoning, action) triples.** `parseTranscript` should canonicalize to the (o, r, a) triple shape rather than custom per-slice formats. Doing so unlocks five "free" competency scores from the literature — backtracking & exploration, task decomposition, observation reading, self-verification, objective quality — without slice-specific code. Eagerness (slice-3) and destructive-action (slice-2) probes both benefit directly from the self-verification + backtracking axes. ([Ou et al., AgentDiagnose, EMNLP 2025 Demos](https://aclanthology.org/2025.emnlp-demos.15.pdf); see [`research/harness-engineering.md`](research/harness-engineering.md) Source 4)
- **Hand-rolled harness is endorsed at our scale (cross-check on ADR 0008).** Cutting-edge agent-eval literature (HAL, OLMES) does not argue against hand-rolled harnesses for projects at our scale (n=25–100/cell, 1–5 slices/quarter). The consistent message is "your harness is fine, document the configuration manifest." HAL itself is hand-rolled. The framework-vs-hand-rolled debate is settled in favor of hand-rolled-with-manifest. ([Stroebl et al., HAL, ICLR 2026](https://openreview.net/pdf/c74cc98588086d5efd7cf146b47b0d5112ab3f90.pdf); [Gu et al., OLMES](https://aclanthology.org/2025.findings-naacl.282.pdf); see [`research/harness-engineering.md`](research/harness-engineering.md) target 4 verification)
- **Single-line interventions can carry a slice.** QuittingAgents (Oct 2025) added one quit-instruction sentence to the system prompt and got +0.39 mean safety / −0.03 helpfulness across 12 LLMs on the ToolEmu suite (Claude 4 Sonnet: +1.21 safety / −0.01 helpfulness). PropensityBench (Nov 2025) showed tool *names* alone (not descriptions) shift Claude Sonnet 4 and o4-mini behavior substantially. Slice-4 should include cells for (a) a one-line "stop and quit if underspecified" addition to system_prompt and (b) a tool-rename ablation (e.g. `execute_sql` → `execute_sql_dangerous`). These are the lowest-effort highest-ROI interventions for supabase-mcp maintainers. ([QuittingAgents, Oct 2025](https://github.com/victorknox/QuittingAgents); [PropensityBench, Nov 2025](https://arxiv.org/html/2511.20703v1); see [`research/agent-eagerness.md`](research/agent-eagerness.md) Sources 3, 4)
- **Mitigation-aware capability evaluation: SUT = system + mitigation, not bare model.** Anthropic's Sabotage Evals (Oct 2024) measure capability *given the mitigation in place*, because that is the deployment reality. Aligns with `CLAUDE.md`'s anti-pattern of "model-as-SUT framing." This is the academic citation behind the supabase-mcp framing: the SUT is the server-with-its-flags, not the bare model. ([Benton, Wagner et al., Anthropic, Oct 2024](https://assets.anthropic.com/m/377027d5b36ac1eb/original/Sabotage-Evaluations-for-Frontier-Models.pdf); see [`research/agent-eagerness.md`](research/agent-eagerness.md) Source 5)
- **Eval Factsheet header per slice.** Each slice findings doc gets a 1-page header (dataset SemVer + content hash, attacker model, scoring rule, judge model + prompt hash, reproducibility commands). Composes with Datasheets-for-Datasets (Gebru et al., CACM 2021); enables per-slice provenance audits. ([Eval Factsheets, arXiv:2512.04062](https://arxiv.org/html/2512.04062v1); see [`research/dataset-construction.md`](research/dataset-construction.md) Source 12)
- **Pre-slice construct-validity checklist** (Bean's eight recommendations, operationalised). Every slice spec must answer yes/no to: (1) precise definition + scope; (2) confounder list + format-control + parser-validation; (3) sampling strategy + item-quality + sensitivity-variants; (4) reuse documentation + new-vs-original delta; (5) contamination test + held-out set; (6) N + power + uncertainty; (7) qualitative+quantitative error analysis with confounder check; (8) phenomenon→task→metric→claim rationale chain. Full checklist in [`research/construct-validity.md`](research/construct-validity.md) Target 1. ([Bean et al., NeurIPS 2025 D&B](https://openreview.net/pdf/5e50cba825c86cf5d8a7c4148c1a7241d89c15ca.pdf))
- **Multi-prompt floor for rates.** Single-prompt rates are unreliable. Default for slice headlines: report the rate at canonical phrasing *and* the rate band under a paraphrase-perturbation probe (≥3 semantic-preserving rewrites of every scored prompt). If band-width exceeds the effect being claimed, the slice is measuring phrasing, not the construct. ([Habba et al., DOVE, ACL Findings 2025](https://aclanthology.org/2025.findings-acl.611.pdf); [Cohen-Inger et al., C-BOD, EMNLP 2025](https://aclanthology.org/2025.emnlp-main.1098.pdf); see [`research/construct-validity.md`](research/construct-validity.md) Sources 5, 6)
